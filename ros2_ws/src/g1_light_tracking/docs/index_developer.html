Ta wersja dokumentacji jest skierowana do osób rozwijających kod. Zawiera bardziej techniczne omówienie architektury, gotowe przykłady kodu dla ROS2 i Pythona, konwencje implementacyjne, przykłady node’ów, launchy, diagnostyki, pracy headless oraz wzorce bezpiecznego fallbacku.

Docelowe środowisko: Unitree G1 EDU, Ubuntu 22.04, ROS2, praca głównie przez SSH i bez GUI. Projekt należy rozwijać tak, aby podstawowa pętla robota nie zależała od modułów pomocniczych.

Nawigacja

Przegląd techniczny

Repo rozwija warstwowy system robotyczny, w którym kolejne node’y realizują odrębne etapy: percepcję, lokalizację, tracking, logikę logistyczną, misję i sterowanie. Za tor docelowy należy uznawać architekturę modern. Warstwa legacy ma charakter kompatybilnościowy i migracyjny.

Filozofia rozwoju

Core loop ma być prosty, przewidywalny i bezpieczny. Rozszerzenia mają być opcjonalne.

Preferowany interfejs runtime

ROS2 topics, messages, services, actions, parametry i logi terminalowe.

Preferowany interfejs operatorski

Terminal, TUI, topic echo, raporty tekstowe/YAML/JSON. GUI tylko opcjonalnie.

Najważniejsza zasada

Programista przez SSH ma rozumieć nie tylko stan, ale także zamiary robota.

Dla programisty: jeżeli dodajesz nowy moduł, myśl od razu o tym, jak będzie obserwowany przez logi, TUI i topic diagnostics, oraz jak zachowa się, gdy zabraknie danych.

Pipeline i odpowiedzialności

camera / source
  -> perception_node
  -> localization_node
  -> tracking_node
  -> parcel_track_node
  -> mission_node
  -> control_node

Każdy node ma jedną główną odpowiedzialność:

  • perception_node — wyłącznie obserwacja 2D i klasyfikacja źródeł percepcji.
  • localization_node — przejście 2D → 3D i wybór metody estymacji.
  • tracking_node — spójność identyfikatorów w czasie.
  • parcel_track_node — domena logistyczna i kojarzenie QR.
  • mission_node — wybór aktywnego celu i stanu zadania.
  • control_node — generowanie komend ruchu i ograniczenia safety.
Antywzorzec: nie łącz percepcji, trackingu, logiki misji i sterowania w jednym node’zie. To utrudnia debugowanie, zwiększa ryzyko i psuje możliwość modularnej rozbudowy.

Wzorce implementacyjne

Preferowane wzorce

  • jedna główna odpowiedzialność na node,
  • parametry ROS2 zamiast hardcode,
  • jawny lokalny stan przez dataclasses,
  • małe funkcje pomocnicze,
  • czytelne fallbacki,
  • logi opisujące stan i zamiary,
  • opcjonalne moduły odseparowane od core loop.

Antywzorce

  • ukryta komunikacja poza ROS2,
  • core zależny od GUI,
  • crash przy braku opcjonalnego backendu,
  • brak topiców lub logów diagnostycznych,
  • brak wyjaśnialności dla cmd_vel.
# Dobra praktyka: jawny fallback przy braku depth

    self.get_logger().warning(
        "Brak DepthNavHint. Sterowanie działa bez warstwy safety z głębi."
    )
    return twist

Przykład node'a ROS2

Poniżej wzorzec prostego node’a zgodnego ze stylem projektu.

#!/usr/bin/env python3
\"\"\"Node ROS 2 odpowiedzialny za publikację prostego statusu systemu.

Node odbiera stan misji i publikuje skrócony status diagnostyczny.
Moduł nie wpływa na sterowanie robotem i ma charakter obserwacyjny.
\"\"\"

from __future__ import annotations

from dataclasses import dataclass

import rclpy
from rclpy.node import Node
from std_msgs.msg import String
from g1_light_tracking.msg import MissionState


@dataclass
class StatusState:
    \"\"\"Lokalny stan node'a diagnostycznego.\"\"\"

    last_state: str = "-"
    updates: int = 0


class StatusMirrorNode(Node):
    \"\"\"Prosty node publikujący skrócony stan misji na osobnym topicu.\"\"\"

    def __init__(self) -> None:
        super().__init__("status_mirror_node")

        self.declare_parameter("mission_state_topic", "/mission/state")
        self.declare_parameter("status_topic", "/debug/mission_status_text")

        self.mission_state_topic = str(self.get_parameter("mission_state_topic").value)
        self.status_topic = str(self.get_parameter("status_topic").value)

        self.state = StatusState()
        self.pub = self.create_publisher(String, self.status_topic, 10)
        self.create_subscription(MissionState, self.mission_state_topic, self.on_mission_state, 10)

        self.get_logger().info(
            f"StatusMirrorNode started. mission_state_topic={self.mission_state_topic}, "
            f"status_topic={self.status_topic}"
        )

    def on_mission_state(self, msg: MissionState) -> None:
        \"\"\"Aktualizuje lokalny stan i publikuje uproszczony tekst diagnostyczny.\"\"\"

        self.state.last_state = msg.state
        self.state.updates += 1

        out = String()
        out.data = f"state={msg.state} prev={msg.previous_state} updates={self.state.updates}"
        self.pub.publish(out)


def main(args=None) -> None:
    rclpy.init(args=args)
    node = StatusMirrorNode()
    try:
        rclpy.spin(node)
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == "__main__":
    main()

Publisher / Subscriber

Przykład subskrypcji detekcji i publikacji alarmu

from std_msgs.msg import String
from g1_light_tracking.msg import Detection2D

class DetectionAlarmNode(Node):
    \"\"\"Node wykrywający brak detekcji przez dłuższy czas.\"\"\"

    def __init__(self) -> None:
        super().__init__("detection_alarm_node")
        self.pub = self.create_publisher(String, "/debug/detection_alarm", 10)
        self.create_subscription(Detection2D, "/perception/detections", self.on_detection, 50)
        self.last_detection_time = None
        self.timer = self.create_timer(1.0, self.check_timeout)

    def on_detection(self, msg: Detection2D) -> None:
        self.last_detection_time = self.get_clock().now()

    def check_timeout(self) -> None:
        if self.last_detection_time is None:
            alarm = String()
            alarm.data = "Brak jakichkolwiek detekcji od startu node'a."
            self.pub.publish(alarm)
            return

        age = (self.get_clock().now() - self.last_detection_time).nanoseconds / 1e9
        if age > 2.0:
            alarm = String()
            alarm.data = f"Brak świeżych detekcji od {age:.2f}s"
            self.pub.publish(alarm)

Przykład buforowania lokalnego stanu z wielu topiców

from dataclasses import dataclass
from geometry_msgs.msg import Twist
from g1_light_tracking.msg import MissionTarget, DepthNavHint

@dataclass
class ControlInputs:
    latest_target: MissionTarget | None = None
    latest_depth_hint: DepthNavHint | None = None

class ExampleControlNode(Node):
    def __init__(self) -> None:
        super().__init__("example_control_node")
        self.inputs = ControlInputs()
        self.pub = self.create_publisher(Twist, "/cmd_vel", 10)
        self.create_subscription(MissionTarget, "/mission/target", self.on_target, 10)
        self.create_subscription(DepthNavHint, "/navigation/depth_hint", self.on_depth_hint, 10)

    def on_target(self, msg: MissionTarget) -> None:
        self.inputs.latest_target = msg
        self.update_control()

    def on_depth_hint(self, msg: DepthNavHint) -> None:
        self.inputs.latest_depth_hint = msg

    def update_control(self) -> None:
        if self.inputs.latest_target is None:
            return
        twist = Twist()
        # TODO: Rozwinąć pełną logikę sterowania zależną od aktywnego celu.
        self.pub.publish(twist)

Logowanie i diagnostyka

Log ma pomagać operatorowi i programiście pracującemu przez SSH.

Dobre logi startowe

self.get_logger().info(
    f"Perception node started. image_topic={self.image_topic}, "
    f"camera_info_topic={self.camera_info_topic}, "
    f"enable_qr={self.enable_qr}, enable_apriltag={self.enable_apriltag}"
)

Dobre logi zmiany stanu

if new_state != self.current_state:
    self.get_logger().info(
        f"Mission state changed: {self.current_state} -> {new_state}, reason={reason}"
    )
    self.current_state = new_state

Dobre logi fallbacku

if self.apriltag_detector is None:
    self.get_logger().warning(
        "AprilTag backend unavailable. Node działa dalej bez detekcji AprilTag."
    )
Zasada: nie spamuj każdej klatki, ale zawsze loguj: start, zmianę stanu, degradację, włączenie fallbacku, utratę krytycznych danych i powody blokowania ruchu.

Fallback i safety

Brak danych nie może wyglądać jak poprawne działanie. Gdy dane są stare, niespójne albo niebezpieczne, preferowany jest bezpieczny fallback.

Przykład: neutralny cmd_vel przy braku celu

from geometry_msgs.msg import Twist

def make_safe_stop() -> Twist:
    \"\"\"Tworzy neutralną komendę zatrzymania robota.\"\"\"
    return Twist()

if self.latest_target is None:
    self.get_logger().warning(
        "Brak aktywnego celu misji. Publikuję zerowe cmd_vel."
    )
    self.pub.publish(make_safe_stop())
    return

Przykład: depth hint blokuje ruch do przodu

def apply_depth_navigation(self, twist: Twist) -> Twist:
    \"\"\"Modyfikuje komendę ruchu na podstawie lokalnej oceny przeszkód.\"\"\"

    if self.latest_depth_hint is None:
        self.get_logger().warning(
            "Brak DepthNavHint. Sterowanie działa bez warstwy safety z głębi."
        )
        return twist

    if self.latest_depth_hint.obstacle_ahead:
        self.get_logger().warning(
            f"Obstacle ahead. forward_clearance_m="
            f"{self.latest_depth_hint.forward_clearance_m:.2f}. Zeruję ruch do przodu."
        )
        twist.linear.x = 0.0

    return twist

Launchery i integracja

Launchery mają składać system, a nie ukrywać logikę biznesową. Moduły opcjonalne należy uruchamiać warunkowo.

Minimalny launcher z argumentem

from launch import LaunchDescription
from launch.actions import DeclareLaunchArgument
from launch.conditions import IfCondition
from launch.substitutions import LaunchConfiguration
from launch_ros.actions import Node

def generate_launch_description():
    with_debug = LaunchConfiguration("with_debug")

    return LaunchDescription([
        DeclareLaunchArgument("with_debug", default_value="true"),

        Node(
            package="g1_light_tracking",
            executable="mission_node",
            output="screen",
        ),

        Node(
            package="g1_light_tracking",
            executable="debug_node",
            output="screen",
            condition=IfCondition(with_debug),
        ),
    ])

Wzorzec dla modułów opcjonalnych

  • diagnostyka, TUI, replay, recorder i GUI powinny być uruchamiane warunkowo,
  • brak modułu opcjonalnego nie może zepsuć launchu core,
  • launch ma jasno komunikować, które node’y startują.

Headless / no-display

Projekt działa na robocie bez monitora. GUI może być dodatkiem, ale nie może być warunkiem działania core.

Wymagania dla kodu

  • nie zakładaj obecności DISPLAY,
  • nie wymagaj cv2.imshow, Qt ani Matplotlib GUI,
  • preferuj TUI, logi i topic diagnostics,
  • błędy no display powinny kończyć się warningiem i wyłączeniem GUI, a nie crashem core.
import os

if not os.environ.get("DISPLAY"):
    self.get_logger().warning(
        "Brak DISPLAY. Przechodzę w tryb headless i wyłączam wizualizację GUI."
    )
    self.enable_gui = False

Checklisty developerskie

Przed dodaniem nowego node’a

  1. Czy ma jedną główną odpowiedzialność?
  2. Czy wejścia/wyjścia są jawne i idą przez ROS2?
  3. Czy brak danych prowadzi do przewidywalnego fallbacku?
  4. Czy logi tłumaczą stan i decyzje?
  5. Czy działa przez SSH i headless?
  6. Czy da się go wydzielić do osobnej paczki, jeśli urośnie?

Przed zmianą logiki sterowania

  1. Czy operator będzie rozumiał, dlaczego robot jedzie lub stoi?
  2. Czy ruch jest blokowany bezpiecznie przy starych lub złych danych?
  3. Czy cmd_vel ma ścieżkę diagnostyczną?
  4. Czy fallback publikuje bezpieczną wartość neutralną?

Przed mergem

  1. Czy komentarze i docstringi po polsku nadal są zgodne z kodem?
  2. Czy TODO zostały zachowane lub zaktualizowane?
  3. Czy launch, README i docs nie rozjechały się z implementacją?
  4. Czy build i podstawowe smoke testy przechodzą?
Najważniejsza zasada developerska: kod ma być nie tylko poprawny technicznie, ale też zrozumiały, obserwowalny i bezpieczny operacyjnie na robocie działającym headless.
