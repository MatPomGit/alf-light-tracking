Nawigacja

Przegląd

Repozytorium zawiera warstwowy system dla robota i środowisk testowych. Aktualny tor nowoczesny prowadzi od obrazu przez detekcję 2D, lokalizację 3D, tracking, logikę przesyłek i misji aż do uproszczonego sterowania. Równolegle dostępny jest tor legacy, który może działać samodzielnie albo zasilać nowy pipeline przez adapter.

Nowoczesny pipeline

perception → localization → tracking → parcel_track → mission → control

Rozszerzenia

depth_mapper_node, visual_slam_node, debug_node, topdown_odom_viewer_node

Legacy compatibility

Detektor światła, bridge'e Unitree, replay CSV, turtlesim, adapter JSON → nowe msg.

Główny punkt wejścia

launch/unified_system.launch.py z trybami modern, legacy, hybrid.

Architektura i przepływ danych

RGB image
  -> perception_node
    -> /perception/detections (Detection2D)
      -> localization_node
        -> /localization/targets (LocalizedTarget)
          -> tracking_node
            -> /tracking/targets (TrackedTarget)
              -> parcel_track_node
                -> /tracking/parcel_bindings
                -> /tracking/parcel_tracks (ParcelTrack)
                  -> mission_node
                    -> /mission/state
                    -> /mission/target
                      -> control_node

Równolegle mogą działać dodatkowe źródła danych i diagnostyka:

  • depth_mapper_node publikuje DepthNavHint dla sterowania.
  • visual_slam_node publikuje estymację ruchu i ścieżkę.
  • topdown_odom_viewer_node wizualizuje odometrię, tracki i cele.
  • legacy_detection_adapter_node wstrzykuje dane z toru legacy do nowego modelu wiadomości.

Launchery i komendy

Wszystkie poniższe polecenia wykonuj z katalogu ros2_ws/ po wcześniejszym buildzie i po source install/setup.bash.

Legacy i adapter

Warstwa legacy została zachowana obok nowoczesnej architektury. Nie nadpisuje obecnych funkcjonalności. Jej rola to kompatybilność ze starszym projektem, integracje sprzętowe i środowiska demonstracyjne. 

light_spot_detector_node or csv_detection_replay_node
  -> /light_tracking/detection_json
    -> legacy_detection_adapter_node
      -> /perception/detections
      -> optionally /tracking/targets
        -> mission_node
          -> control_node

W trybie legacy adapter może publikować także przybliżone TrackedTarget. W trybie hybrid publikuje tylko Detection2D, żeby nie dublować tracków z tracking_node.

Jak scalono repozytoria i co dalej

Scalenie zostało wykonane zachowawczo. Aktualne repozytorium pozostało bazą docelową, a elementy starszego projektu zostały dołożone jako osobna warstwa kompatybilności. Dzięki temu nie nadpisano gotowych node'ów, wiadomości, konfiguracji ani launcherów z obecnego pakietu.

ObszarJak został scalonyDlaczego tak
Node'y nowoczesnePozostawione bez zastępowania w ../g1_light_tracking/nodes/.To jest docelowy pipeline produkcyjny i nie wolno było utracić istniejących funkcjonalności.
Node'y legacyDodane obok jako osobne entrypointy w katalogu ../g1_light_tracking/.Pozwala to uruchamiać stare źródła danych bez przepisywania wszystkiego jednocześnie.
KonfiguracjePliki starszego stosu dostały prefiks legacy_* w katalogu ../config/.Uniknięto nadpisywania aktualnych parametrów i rozjazdów nazw.
Integracja danychDodano legacy_detection_adapter_node i logikę w ../g1_light_tracking/utils/legacy_adapter.py.Adapter tłumaczy JSON na Detection2D i opcjonalnie TrackedTarget, więc nowa logika może konsumować stare wejścia.
UruchamianieDodano ../launch/unified_system.launch.py oraz launchery pomocnicze legacy.Jeden punkt wejścia upraszcza migrację, testy A/B i tryb hybrydowy.
Zależności sprzętoweMosty Unitree i kamera legacy są opcjonalne oraz uruchamiane warunkowo.Repo działa też bez pełnego środowiska sprzętowego, co ułatwia rozwój i CI.

Rzeczywista sekwencja merge'u

  1. Utrzymano aktualny model wiadomości z katalogu ../msg/ jako jedyny kontrakt docelowy.
  2. Przeniesiono starsze node'y jako warstwę kompatybilności, zamiast podmieniać obecne moduły.
  3. Dodano osobne pliki legacy_*.yaml, żeby stare parametry nie nadpisały nowego strojenia.
  4. Dodano adapter legacy JSON → nowe wiadomości oraz tryby legacy i hybrid.
  5. Wpięto legacy źródła do nowego mission_node i control_node przez launcher unified.
  6. Rozbudowano dokumentację tak, żeby każda ścieżka i komenda odpowiadała aktualnej strukturze repo.

Dalsze kroki, aby w pełni przejść na aktualne repozytorium

  1. Przestać traktować JSON jako kontrakt główny. Wszystkie nowe integracje powinny publikować od razu Detection2D albo LocalizedTarget, a adapter legacy powinien zostać tylko przejściowym mostem.
  2. Wygasić dublujące źródła tracków. Docelowo tylko tracking_node powinien publikować /tracking/targets; tryb hybrydowy jest po to, aby testować migrację bez równoległych ID z legacy.
  3. Przenieść heurystyki starego detektora do nowego etapu percepcji. Najlepszym miejscem jest ../g1_light_tracking/nodes/perception_node.py, a nie osobny tor JSON.
  4. Zastąpić bridge'e sprzętowe spójnymi adapterami ROS 2. Mosty Unitree i legacy kamery powinny docelowo publikować te same topiki, które zużywa nowoczesny pipeline.
  5. Domknąć build i testy integracyjne. Poza pytest trzeba dodać testy launch, smoke-test colcon build i scenariusze dla trybów modern, legacy, hybrid.
  6. Oznaczyć komponenty do deprecacji. Gdy nowe źródła danych osiągną pełne pokrycie funkcjonalne, legacy node'y powinny zostać oznaczone jako deprecated w README i launcherach.
  7. Zmienić domyślną ścieżkę operacyjną zespołu. Dokumentacja, instrukcje wdrożeniowe i skrypty developerskie powinny wskazywać unified_system.launch.py mode:=modern jako jedyny tryb domyślny.

Praktycznie: pełna migracja jest zakończona dopiero wtedy, gdy legacy_detection_adapter_node, light_spot_detector_node i pozostałe bridge'e nie są już potrzebne do codziennego uruchamiania, a wszystkie wejścia systemu trafiają bezpośrednio do obecnego modelu wiadomości.

Tryby unified

Launcher ../launch/unified_system.launch.py przełącza cały system między trzema trybami.

TrybCo uruchamiaKiedy używać
modernpełny nowoczesny pipeline + depth + visual SLAM + debugDomyślne uruchomienie systemu.
legacylegacy detector + adapter + nowy mission_node i control_nodeIntegracja ze starym źródłem danych.
hybridpełny nowoczesny pipeline plus legacy detector jako dodatkowe źródło Detection2DStrojenie i migracja krok po kroku.
 
ros2 launch g1_light_tracking unified_system.launch.py mode:=modern
ros2 launch g1_light_tracking unified_system.launch.py mode:=legacy
ros2 launch g1_light_tracking unified_system.launch.py mode:=hybrid
ros2 launch g1_light_tracking unified_system.launch.py mode:=hybrid with_legacy_camera:=true
ros2 launch g1_light_tracking unified_system.launch.py mode:=legacy with_unitree_bridges:=true

Wersjonowanie

Hook wersjonowania można zainstalować z trzech poziomów katalogów:

# z katalogu głównego repo
bash install_git_hooks.sh

# z katalogu ros2_ws/
bash install_git_hooks.sh

# z katalogu ros2_ws/src/g1_light_tracking/
bash scripts/install_git_hooks.sh

Każdy wariant instaluje hook pre-commit, który uruchamia scripts/version_bump.py i aktualizuje VERSION, setup.py oraz package.xml.
