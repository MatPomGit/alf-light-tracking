Nawigacja

Przegląd

Projekt implementuje warstwowy pipeline dla percepcji obrazu, lokalizacji 3D, trackingu, logiki misji i sterowania. W repo istnieje także warstwa kompatybilności legacy, ale za tor docelowy należy uznawać architekturę modern. System ma być rozwijany tak, aby robot mógł działać autonomicznie, a programista przez SSH zawsze rozumiał aktualny stan i zamiary robota.

Cel systemu

Bezpieczne i obserwowalne sterowanie robotem na podstawie percepcji, trackingu i logiki misji.

Preferowany runtime

unified_system.launch.py mode:=modern lub prod.launch.py.

Preferowane narzędzia

Terminal, ROS2 CLI, TUI, logi, topic echo, raporty tekstowe i YAML/JSON.

Ważne ograniczenie

Brak monitora i praca headless oznacza, że funkcje GUI muszą być opcjonalne.

Najważniejsza zasada operacyjna: robot ma działać autonomicznie, ale człowiek przez SSH ma zawsze rozumieć: co robot widzi, co śledzi, jaki jest stan misji, dlaczego publikuje dane sterowanie i czego systemowi brakuje.

Architektura i przepływ danych

RGB image
  -> perception_node
    -> /perception/detections (Detection2D)
      -> localization_node
        -> /localization/targets (LocalizedTarget)
          -> tracking_node
            -> /tracking/targets (TrackedTarget)
              -> parcel_track_node
                -> /tracking/parcel_bindings
                -> /tracking/parcel_tracks (ParcelTrack)
                  -> mission_node
                    -> /mission/state
                    -> /mission/target
                      -> control_node
                        -> /cmd_vel

Równolegle mogą działać moduły pomocnicze:

  • depth_mapper_node publikuje DepthNavHint do warstwy safety / sterowania.
  • visual_slam_node publikuje odometrię i ścieżkę do diagnostyki.
  • debug_node loguje najważniejsze topiki pipeline’u.
  • tui_monitor_node prezentuje stan systemu w terminalu.
  • legacy_detection_adapter_node wstrzykuje dane z toru legacy do nowego kontraktu wiadomości.
Zasada projektowa: moduły pomocnicze mają wspierać diagnostykę i bezpieczeństwo, ale ich brak lub awaria nie mogą zatrzymywać głównej pętli robota.

Diagnostyka i observability

Projekt ma być maksymalnie obserwowalny przez terminal i topiki diagnostyczne. Operator powinien móc sprawdzić nie tylko „czy system działa”, ale także co robot aktualnie robi i dlaczego.

Podstawowe komendy diagnostyczne

cd ros2_ws
source install/setup.bash
ros2 topic list
ros2 topic echo /perception/detections
ros2 topic echo /tracking/targets
ros2 topic echo /mission/state
ros2 topic echo /mission/target
ros2 topic echo /navigation/depth_hint
ros2 topic echo /cmd_vel

Co powinno być obserwowalne

  • świeżość danych wejściowych i topiców,
  • liczba detekcji i tracków,
  • aktywny stan misji i powód jego zmiany,
  • czy depth layer ogranicza ruch,
  • czy komendy cmd_vel są publikowane, blokowane lub modyfikowane,
  • czy backendy opcjonalne, np. QR albo AprilTag, są aktywne.
Ważne: logi, TUI i topici diagnostyczne nie są dodatkiem estetycznym. Są częścią zdolności zespołu do diagnozowania problemów na robocie działającym przez SSH.

Headless / SSH / no-display

Robot nie ma monitora, dlatego wszystkie kluczowe ścieżki runtime muszą działać bez GUI. Zakładaj, że operator pracuje przez SSH i ma do dyspozycji:

  • terminal,
  • ros2 launch, ros2 run, ros2 topic echo,
  • TUI,
  • logi stdout/stderr,
  • raporty tekstowe / YAML / JSON.

Czego nie wolno zakładać

  • obecności DISPLAY,
  • obecności środowiska desktopowego,
  • możliwości użycia cv2.imshow,
  • jakiegokolwiek GUI jako warunku działania systemu core.
# Typowa diagnoza po SSH
cd ros2_ws
source install/setup.bash
ros2 launch g1_light_tracking unified_system.launch.py mode:=modern

# drugi terminal
cd ros2_ws
source install/setup.bash
ros2 run g1_light_tracking tui_monitor_node

Unitree i wdrożenie

Docelowym środowiskiem jest Unitree G1 EDU z Ubuntu 22.04 i środowiskiem klasy Jetson. Warstwa integracyjna z robotem ma być odseparowana od core pipeline’u: awaria bridge’a sprzętowego nie może psuć percepcji, trackingu ani logiki misji.

Minimalna checklista wdrożeniowa

  1. Zainstalować zależności systemowe i Pythonowe.
  2. Zbudować workspace z ros2_ws/.
  3. Sprawdzić importy i backendy opcjonalne.
  4. Uruchomić pipeline w trybie bezpiecznym bez realnego ruchu.
  5. Zweryfikować topici: percepcja, tracking, misja, depth hint, cmd_vel.
  6. Dopiero potem włączyć bridge sprzętowy Unitree.
# Legacy + bridge Unitree
cd ros2_ws
source install/setup.bash
ros2 launch g1_light_tracking unified_system.launch.py mode:=legacy with_unitree_bridges:=true
Bezpieczny start: najpierw sprawdź, czy system poprawnie liczy i publikuje cmd_vel, a dopiero potem podłączaj fizyczne wysyłanie komend do robota.

Legacy i adapter

Warstwa legacy została zachowana, ale nie jest już architekturą docelową. Jej rola to:

  • kompatybilność ze starym detektorem,
  • integracja z istniejącymi źródłami JSON,
  • replay danych historycznych,
  • scenariusze demonstracyjne i migracyjne.
light_spot_detector_node / csv_detection_replay_node
  -> /light_tracking/detection_json
    -> legacy_detection_adapter_node
      -> /perception/detections
        -> nowy pipeline mission / control

W trybie hybrid legacy ma dostarczać jedynie dodatkowe wejścia do nowego modelu wiadomości, a nie narzucać starego kontraktu jako źródła prawdy dla całego systemu.

Scalenie repo i migracja

Scalenie zostało wykonane zachowawczo: obecny pakiet i jego wiadomości pozostały bazą docelową, a elementy starszego projektu zostały dołączone jako warstwa kompatybilności.

ObszarJak został scalonyDlaczego tak
Node’y nowoczesnePozostawione jako główna ścieżka runtime.To docelowy pipeline produkcyjny.
Node’y legacyDodane obok jako entrypointy kompatybilnościowe.Umożliwia migrację bez przepisywania wszystkiego naraz.
KonfiguracjeLegacy dostało osobne pliki legacy_*.yaml.Uniknięto nadpisywania strojenia nowego pipeline’u.
Integracja danychDodano adapter JSON → nowe message.Nowa logika może konsumować stare źródła bez zmiany kontraktu wiadomości.

Dalsze kroki do pełnej migracji

  1. Traktować obecny model wiadomości jako jedyne źródło prawdy.
  2. Wygaszać dublujące źródła tracków w warstwie legacy.
  3. Przenosić heurystyki starej percepcji do nowego perception_node.
  4. Wydzielać rozszerzenia i narzędzia do osobnych paczek, gdy przestają być lokalne dla core.
